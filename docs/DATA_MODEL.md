# Data Model

```
seeds/*.csv   ──►  raw.*   ──►  bronze.*   ──►  silver.*   ──►  gold.customer_360
                              (append)        (merge)         (table)
```

## Layers

| Layer  | Materialization                  | Role                                                  |
| ------ | -------------------------------- | ----------------------------------------------------- |
| bronze | `incremental` + `append`         | Durable typed landing. Append-only history; never modify rows. Customers append a daily snapshot. |
| silver | `incremental` + `merge` on key   | Current state per natural key. Reads only new bronze rows, dedupes to latest version, MERGEs. Liquid-clustered on join keys. |
| gold   | `table` (full rebuild)           | `customer_360` plus ephemeral `int_*` rollups. Partitioned by `customer_segment`. |

### Why each materialization

* **Bronze append** — preserves the full source history; silver/gold can
  always be rebuilt from it.
* **Silver merge** — runtime stays proportional to *new* bronze data, not
  total volume, while still giving downstream a clean "current state".
* **Gold full rebuild** — at <1M customers it's seconds. Switch to
  incremental once volume hits ~10M.

## Conventions

* `bronze_*`, `silver_*` prefix matches the folder.
* `int_*` for ephemeral gold building blocks.
* `is_<adjective>` for boolean flags.
* Rolling-window columns name the window: `txns_last_90d`.

The autogenerated `dbt docs serve` site has the full column-level
dictionary and lineage graph.
